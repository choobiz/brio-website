## SR&ED Project 1: Empirical Characterization of GoHighLevel Inbound-Webhook Routing Failure Modes

*Mapped from: Task 2 in work report 2026-04-26.md*

### 1. Technological Uncertainty (Form T661, Line 242)

**Existing knowledge base:**
The team operates a multi-stage CRM lead pipeline built on GoHighLevel (GHL), a closed-source SaaS workflow engine reached via inbound HTTPS webhooks. Standard practice for HTTP webhook integrations relies on the response status code as the authoritative success signal: `2xx` indicates successful receipt and processing, `4xx`/`5xx` indicate failure. This is the convention codified in Stripe's webhook spec, the W3C HTTP semantics RFC 9110, and every mainstream payment/messaging webhook integration encountered to date.

**Technological uncertainty:**
A class of submissions to the GHL inbound webhook were silently failing — the `fetch` call returned `HTTP 200` with a JSON success payload (`{"status":"Success: test request received"}`), yet no contact record was being created in the CRM, no workflow enrollment occurred, and no error surfaced anywhere reachable from the client. It was unknown whether (a) the failure was a transient GHL incident, (b) the workflow's internal action graph was rejecting the payload silently after acceptance, (c) the payload schema had drifted from what the workflow expected, or (d) the webhook trigger identity itself had become disconnected from the workflow despite continuing to return `2xx` responses. None of these failure modes are documented in GHL's public API reference (the documentation only states that `2xx` = success).

**Why standard practice was insufficient:**
Standard webhook-integration debugging — assert `response.ok`, log the response body, retry on non-2xx — had already been implemented in `lib/ghl.ts` and produced no actionable signal. A direct `curl` reproduction returned the same `200` with no diff in body shape vs. a successful enrollment from earlier in the day. There was no existing literature, GHL forum thread, or third-party diagnostic tool that documented the differentiation between a "live" and "stale" GHL inbound webhook, both of which return `2xx`.

### 2. Systematic Investigation (Form T661, Line 244)

**Hypothesis/Approach:**
We hypothesized that the GHL inbound-webhook acceptance layer is decoupled from the workflow trigger layer — i.e., GHL's edge gateway accepts and acknowledges any well-formed POST against any webhook path it has *ever* served, and only the trigger-bound workflow tier silently drops requests whose trigger UUID has been deprecated. If true, the response *body* (not status code) should differ between live and stale endpoints, because the stale endpoint never reaches the trigger-execution server that emits an enrollment ID.

**Work performed:**
1. **Baseline reproduction.** Issued a structured POST against the in-code webhook URL with a synthetic contact payload. Captured `HTTP 200 {"status":"Success: test request received"}`. No error, no `id` field.
2. **Cross-checked workflow side effects.** Inspected GHL → Automation → Workflows → Enrollment History. Three enrollments visible from earlier in the day (timestamps 9:16, 9:16, 9:24), all `Finished`. The synthetic POST was absent. This eliminated hypothesis (a) "transient incident" — the workflow itself was receiving and processing other requests.
3. **Captured the live trigger URL from GHL's UI** (not the cached one in code) and diffed against the in-code value. Same location prefix `7Z5Zm3czsuWjz4zwmsFr`, divergent trigger UUID (`d8c59051-…` in code vs. `cfa234ea-…` in GHL UI). This confirmed identity drift between code and the published workflow trigger — supporting hypothesis (d).
4. **Reproduced against the new URL.** Captured `HTTP 200 {"status":"Success: request sent to trigger execution server","id":"FgSShkJEefXj2TMLl3PA"}`. Two new observable signals: (i) the success-message wording shifted from "test request received" to "request sent to trigger execution server"; (ii) an `id` field appeared, presumably the enrollment ID issued by the trigger-execution tier.
5. **Confirmed enrollment downstream.** Refreshed Enrollment History; the synthetic contact `NewURL WebhookTest` appeared. Confirmed the body-wording differential is the externally-observable signature of a live trigger.
6. **Documented the failure-mode signature** in a long-lived feedback memory (`feedback_ghl_webhook_url_rotates.md`) so the diagnostic doesn't have to be rediscovered. Updated four code locations across two repositories with the live URL.

### 3. Technological Advancement (Form T661, Line 246)

**Advancement achieved:**
- Established that GHL's inbound-webhook stack has an undocumented two-tier architecture: an *edge acceptance* tier that returns `2xx` for any historically-valid trigger UUID under a location, and a *trigger-execution* tier that selectively dispatches only currently-published trigger UUIDs to workflows. The two tiers are externally distinguishable only by the response body wording and the presence/absence of an `id` field.
- Identified the operational invariant: **publishing a workflow in the GHL UI rotates the trigger UUID**, silently invalidating any client code that has cached the prior URL. No deprecation header, no warning email, no `4xx` is emitted for the stale URL — the failure is fully invisible to standard webhook-monitoring stacks.
- Derived a reusable defensive pattern: when the response body contains `"test request received"` (rather than `"request sent to trigger execution server"`) and lacks an `id` field, treat the URL as compromised and rotate. This pattern is generalizable to any GHL integration, not just the Brio one.

**Residual uncertainty:**
- Unknown whether GHL rotates the trigger UUID on *every* publish, or only on certain trigger-config edits (e.g., adding/removing payload-mapping fields). Future investigation would require deliberately re-publishing with no edits and observing whether the UUID changes — deferred until needed.
- Unknown whether a future version of GHL might return `4xx` for stale URLs (which would render the body-wording check unnecessary). Solution remains forward-compatible because checking `response.ok` is still primary.

## SR&ED Project 2: Design and Validation of a Multi-Variant Lead-Capture Form Compatible with GHL Custom-Field Encoding Constraints

*Mapped from: Task 3 in work report 2026-04-26.md*

### 1. Technological Uncertainty (Form T661, Line 242)

**Existing knowledge base:**
The team has a shared `LeadForm` React component used across 14+ landing pages and a city-page layout. The downstream CRM (GoHighLevel) uses custom-field types — `Single Line Text`, `Single Option Dropdown`, `Multi Option Dropdown`, `Address`, `Date` — each with case-sensitive value-validation rules that are not exposed via the public API. Prior incidents (commits `cc0c621` and `e20b7be`) had already established that GHL silently rejects payload values whose case or formatting does not match the dropdown option strings configured in the GHL UI, and that the `Address` field type rejects free-form strings (must be `Single Line` for arbitrary input).

**Technological uncertainty:**
The form needed to extend its payload to populate four GHL custom fields (`estimated_budget`, `project_start_date`, `financing_interest`, `sms_opt_in`) without (a) breaking submission for any of 14+ existing landing pages that pass differing service-bucket props, (b) introducing required-field friction that would reduce conversion, and (c) triggering the same case-sensitivity rejection that previously caused the `cc0c621` incident. It was not known whether a single shared component could simultaneously support service-specific budget brackets, a TCPA-compliant implicit-consent pattern (no checkbox), a "compact" variant that hides labels, and a "stacked" variant with full labels — without per-page configuration changes.

**Why standard practice was insufficient:**
Standard form-library patterns (React Hook Form, Formik) handle field-level validation but do not address downstream CRM-encoding constraints. Naïve approaches considered and rejected:
- Sending raw field slugs (`"unsure"`, `"under-25k"`) — rejected because GHL dropdown-option matching is case- and string-sensitive against full-text labels, per the `cc0c621` learning.
- Hardcoding service-specific budget brackets per page — rejected because it would require touching 14+ pages and introduces drift risk.
- Making budget/start-date required — rejected because user explicitly required no-friction defaults.
- Replacing the SMS-opt-in checkbox with a separate consent gate — rejected because TCPA implicit-consent via adjacent disclaimer is a more conversion-friendly pattern but had no in-house precedent.

### 2. Systematic Investigation (Form T661, Line 244)

**Hypothesis/Approach:**
We hypothesized that an optional `budgetOptions` prop with a sensible 6-bucket default, combined with full-text values matching the GHL custom-field options exactly, would satisfy all 14+ pages without per-page edits *and* allow future per-service overrides without component changes. We further hypothesized that an implicit-consent disclaimer adjacent to the submit button would satisfy TCPA's "express written consent" standard while removing one click from the conversion path.

**Work performed:**
1. **Surveyed the marketing-hub form** (`brio-marketing-hub/src/components/LeadForm.tsx`) — a much more elaborate multi-step form with per-campaign budget options — to extract the canonical full-text label strings GHL accepts. Confirmed labels like `"Within 1 Month"` (not `"1month"`) are required after the `cc0c621` fix.
2. **Designed the value/label split** for the new selects: chose to send full-text values (e.g., `value="Not sure yet"`) rather than slugs to eliminate the case-mismatch class of failures. Documented the choice in code comments.
3. **Designed dual-variant rendering** with shared option arrays: the `stacked` variant uses standalone labels above selects, the `compact` variant prefixes the default option text with a category hint (`"Budget — Not sure yet"`) so the field meaning is discoverable without a label. This was a non-obvious design call — naive approach would have used `<option disabled>` placeholders, which would force users who don't know to make a deliberate "I don't know" selection.
4. **Mapped boolean → string at the payload boundary** rather than at the form layer: `financing_interest: payload.financing_interest ? "Yes" : "No"` and `sms_opt_in: payload.sms_opt_in === false ? "No" : "Yes"`. The asymmetric SMS encoding (defaults to `"Yes"` when undefined) was deliberate — implicit consent is the default unless a future caller explicitly opts out, which keeps the door open for a future explicit-consent form variant without touching the helper.
5. **Verified the design end-to-end** by submitting a real form from the live `kitchen-remodeling` page after deploy. Confirmed: contact created in GHL, all four custom fields populated, no rejection, financing checkbox surfaced as `"Yes"`, SMS opt-in surfaced as `"Yes"` despite no checkbox in the UI.

**Evidence artifacts:**
- Commit: `dc8f43f` (brio-website)
- Files: `brio-website/components/shared/LeadForm.tsx`, `brio-website/lib/ghl.ts`
- Verification: `npx tsc --noEmit` clean exit; live form submission appearing in GHL Enrollment History with all four custom fields populated.

### 3. Technological Advancement (Form T661, Line 246)

**Advancement achieved:**
- Established a reusable pattern for shared lead-capture components that bridge a single React surface to a closed-source CRM with case-sensitive custom-field encoding: send full-text labels as values, encode booleans → enumerated strings at the helper boundary (not the form), and treat optional dropdown defaults as real values (`"Not sure yet"`) rather than placeholder sentinels.
- Validated that an implicit-consent SMS pattern (disclaimer adjacent to submit, no checkbox) integrates cleanly with the GHL `Yes`/`No` dropdown encoding and survives across both label-bearing and label-less UI variants.
- Demonstrated that a single shared component can support service-specific configuration via an optional prop with a sensible default, eliminating the per-page-edit failure mode that affected the prior 14-page rollout.

**Residual uncertainty:**
- Per-service budget bracket overrides have not yet been wired up at any individual page — the default bracket may produce noisier-than-ideal CRM segmentation for high-end services (laneway/spec-home) where the `$250K+` bucket aggregates too much. Deferred until reporting needs surface.
