## Task 2: Diagnose + Fix Dead GHL Webhook URL

### Problem
User reported "I tested the form and didn't see anything get inputted into the GHL." Last successful GHL contact insertion was 9:30 AM that morning; subsequent submissions returned no error to the user but never appeared in GHL.

### Root Cause
The hardcoded inbound webhook trigger UUID in code (`d8c59051-0df7-49bc-a6e7-8aee4735ec38`) was a stale URL from before the Speed-to-Lead workflow was re-published in GHL. After re-publish, GHL rotated the trigger UUID to `cfa234ea-5393-400a-9d25-d6b874bd4fad`. The old URL still returned `HTTP 200 {"status":"Success: test request received"}` — making it appear to work — but no enrollment was created. The new URL returns a different success message (`{"status":"Success: request sent to trigger execution server","id":"..."}`), which is the only reliable signal that the URL is live.

This is a meaningful gotcha worth remembering: GHL inbound webhooks **never** return a 4xx for a stale trigger ID. The 200 response wording is the only differentiator visible from the outside; the authoritative signal is the workflow's Enrollment History tab.

### Diagnostic Path

1. Curled the hardcoded URL from `lib/ghl.ts`:
```bash
curl -X POST 'https://services.leadconnectorhq.com/hooks/7Z5Zm3czsuWjz4zwmsFr/webhook-trigger/d8c59051-0df7-49bc-a6e7-8aee4735ec38' \
  -H 'Content-Type: application/json' \
  -d '{"firstName":"WebTest","lastName":"Diagnostic","phone":"+16041234567","email":"webtest-diag@example.com","source":"webhook-curl-test","tags":["src-webhook-curl-test"]}'
# → HTTP 200 {"status":"Success: test request received"}
```
   The "test request received" wording was the first hint — a live webhook returns "request sent to trigger execution server".

2. User opened GHL → Automation → Workflows → Enrollment History tab. Three enrollments visible from earlier in the day (`amir deliverabilitytest` 9:24am, `basementchain test` 9:16am, `bathroomchain test1` 9:16am), all `Finished`. Critically: my curl test was NOT in the list. This confirmed the workflow's enrollments stopped after 9:24am.

3. Asked user to copy the webhook URL from the workflow's trigger node. They returned:
```
https://services.leadconnectorhq.com/hooks/7Z5Zm3czsuWjz4zwmsFr/webhook-trigger/cfa234ea-5393-400a-9d25-d6b874bd4fad
```
   Same `7Z5Zm3czsuWjz4zwmsFr` location ID, but a different trigger UUID — confirming the rotation theory.

4. Curled the new URL and got `{"status":"Success: request sent to trigger execution server","id":"FgSShkJEefXj2TMLl3PA"}`. The differing success messages and the presence of an `id` field on the new URL were the conclusive proof.

5. User refreshed the Enrollment History tab and confirmed the test contact `NewURL WebhookTest` from my curl appeared.

### Solution

#### File: `brio-website/lib/ghl.ts`

**Change: GHL_WEBHOOK_URL constant**

## Task 3: Add Budget / Start-Date / Financing Fields to Shared LeadForm

### Problem
The shared `LeadForm` component (`components/shared/LeadForm.tsx`) only collected `name / phone / email / address / description`. Four GHL custom fields the workflows expect were always sent as empty strings:
- `estimated_budget`
- `project_start_date` / `preferred_start_date`
- `financing_interest`
- `sms_opt_in`

This meant Workflow #9 (urgency/financing tagger) had no inputs to branch on, and the contact record in GHL had blank columns for budget and timeline. User wanted these fields added with three constraints:
1. SMS opt-in must NOT be a checkbox — replace with implicit consent disclaimer under the submit button to reduce friction.
2. Budget and Start Date should be pre-selected to "unknown" so users who don't know don't have to interact with them.
3. Budget and Start Date must NOT be marked required.

### Solution

#### File: `brio-website/lib/ghl.ts`

**Change 1: Extend `LeadPayload` interface**
- Location: lines 7–18
```ts
export interface LeadPayload {
  name: string;
  phone: string;
  email: string;
  project_address?: string;
  description?: string;
  source: string;
  service?: string;
  city?: string;
  campaign?: string;
  page_url?: string;
  estimated_budget?: string;
  project_start_date?: string;
  financing_interest?: boolean;
  /** Implicit consent via submit-button disclaimer. Defaults to true. */
  sms_opt_in?: boolean;
}
```

**Change 2: Pass field values through the body in `submitLead()`**
- Location: inside the `body` object literal in `submitLead()`
```ts
// Before
estimated_budget: "",
project_start_date: "",
preferred_start_date: "",
financing_interest: "",
sms_opt_in: "",

// After
estimated_budget: payload.estimated_budget ?? "",
project_start_date: payload.project_start_date ?? "",
preferred_start_date: payload.project_start_date ?? "",
financing_interest: payload.financing_interest ? "Yes" : "No",
// Implicit SMS consent: form button disclaimer covers TCPA. Defaults to true.
sms_opt_in: payload.sms_opt_in === false ? "No" : "Yes",
```
- Why: `sms_opt_in === false` keeps the explicit-opt-out path open for future forms, while undefined defaults to `"Yes"` to match the implicit-consent pattern.
- Why: `preferred_start_date` is kept as an alias for `project_start_date` because some pre-existing GHL workflow nodes still reference the legacy field name.

#### File: `brio-website/components/shared/LeadForm.tsx`

This file was rewritten end-to-end (Write, not Edit) because the changes touched the props interface, both render variants, and the submit handler. Critical specifics:

### Why these specific design decisions
- **Pre-selected "Not sure yet" defaults**: User explicitly requested no assumptions and no required fields for budget/start-date. The GHL custom field accepts the literal string `"Not sure yet"` so this is a real value, not a placeholder.
- **`budgetOptions` as an optional prop**: Different services (kitchen vs bathroom vs basement) have different real-world budget brackets. The default 6-bucket range is broad enough to cover all current 14+ pages without per-page configuration, but pages that want service-specific brackets can pass `budgetOptions={[…]}` later without further changes to the component.
- **Implicit SMS consent**: TCPA compliance is satisfied by the disclaimer adjacent to the affirmative click action ("By clicking, you agree…"). User wanted one less interaction step.
- **`financing_interest` mapped boolean → "Yes"/"No"** (not just sent as boolean): GHL custom field is configured as a dropdown with options `Yes` / `No`; case matters per a fix made earlier in commit `cc0c621` of brio-marketing-hub.

### Dependencies & Side Effects
- All 14+ service pages (`app/*/page.tsx`) and `components/city/CityPageLayout.tsx` use the shared `LeadForm` and inherit these changes automatically — no per-page edits needed.
- Future per-service budget brackets can be added by passing `budgetOptions` from each page's `<LeadForm />` invocation.
- The 6-bucket default may produce noisier-than-ideal segmentation if the user later wants to configure GHL smart-lists by exact bracket; recommended to revisit if reporting needs exact ranges per service.
